Verdict

Impressive core,
needs structure

Feature-rich but hard to extend

  • Physics engine is genuinely good

    Binary-search speed-from-power solver, first-order lag smoothing (τ=2.7s), smoothed grade over an 85m window — far more sophisticated than typical indoor apps.

  • Binary FIT export from scratch

    Fully hand-written FIT encoder (CRC, semicircle coords, message definitions) — no library. That's a lot of work and it works correctly.

  • WebRTC live sharing over Socket.IO

    Full signaling + SimplePeer P2P, live peer markers on the map, color-coded rider list — solid architecture for a side-project.

  • Strava segment matching is smart

    Polyline decoding, projected end distance, avg-deviation scoring, lap-aware chronometer — handles the edge cases that simple distance matching misses.

  • Clean dark design system

    Consistent CSS variable palette, zone-colored power chart per-segment, responsive layout breakpoints — looks polished for a utility app.

  • High  Monolithic single file

    1,328 lines of CSS + HTML + JS in one file. No modules, no separation of concerns. Adding a feature means scrolling endlessly and praying you don't break something 800 lines away.

  • High  Global mutable S object

    The Store/subscribe pattern exists but is bypassed constantly — direct S.xxx = yyy mutations everywhere. Side effects are invisible and hard to trace.

  • Med  No settings persistence

    FTP, LTHR, rider weight, rider name reset on every page load. These are your most personal training parameters — they must survive a refresh.

  • Med  French/English mixing

    The Mapillary panel has French UI strings ("Inactif", "Aucune image Mapillary proche", "Rafraîchissement", "Rayon") mixed into an otherwise English UI. Pick one.

  • Med  Live sharing needs a local server

    Default is localhost:8787 — a Socket.IO server the user must run themselves. The feature is invisible to anyone who doesn't know how to spin up a Cloudflare Worker or Node server.

  • Low  No elevation profile

    The map shows the route but there's no elevation profile with current-position indicator. Grade readout is a number — seeing the climb ahead is way more useful mid-ride.

  • Low  No preset routes

    Every ride requires uploading a GPX. Pre-loaded "choix du chef" routes (Ventoux, local Montréal-area classics) would let anyone start riding in seconds.

Implementation: fetch GPX on demand from ./routes/ventoux.gpx using the existing parseGpx() function — no changes needed to the parsing code.

Native ES modules work on GitHub Pages with no build step. Split the 1,328-line file into focused modules.

velox.html (1,328 lines, everything)

velox.html — shell, layout HTML only (~150 lines)
js/state.js — S object, Store, setState(), types
js/bluetooth.js — connectTrainer, connectHrm, parseBike, parseHr
js/physics.js — PhysicsEngine, stepSimulation, getSmoothedGrade
js/gpx.js — parseGpx, parseFit, finalizeRoute, positionAt
js/map.js — drawRoute, updateRouteReadout, Leaflet setup
js/strava.js — segment matching, chronometer, fetchStrava
js/live.js — LiveShare, WebRTC, Socket.IO
js/export.js — buildGPX, buildFIT, download
js/ui.js — updateUINow, drawChart, log, status helpers
css/velox.css — extracted stylesheet
routes/ — choix du chef GPX files
Key move: Each module imports/exports cleanly. velox.html uses <script type="module" src="js/ui.js">. Zero npm, zero build, GitHub Pages serves it natively. The physics and parsing modules can even get unit tests as plain .test.js files if you want them.
